## 二、当前架构设计（最终版）

### 线程分布

```
主线程                         心跳线程池 (8 threads)
┌────────────────────┐         ┌──────────────────────────┐
│ 事件循环            │         │ Thread 0: heartbeat 队列  │
│ call_heart_beat()  │  post   │ Thread 1: heartbeat 队列  │
│  - 仅发送信号       │ ──────→ │ ...                      │
│ set_heart_beat()   │         │ Thread 7: heartbeat 队列  │
│  - post 到对应线程  │         │                          │
│ 玩家命令处理        │         │ process_heartbeats():     │
└────────────────────┘         │  合并队列 → 执行 LPC      │
                               └──────────────────────────┘
```

### 心跳执行流

1. `set_heart_beat(ob, to)` → `pool->thread_for_object(ob)->post(modify_heartbeat)` → 对象加入该线程的私有队列
2. `call_heart_beat()` → 每个线程 `post(process_heartbeats)` → 线程合并 next→main 队列 → 遍历 → 执行 LPC
3. 心跳门控 `g_heartbeat_gate`：创建阶段关闭（跳过 LPC），pool 启动后打开

### 关键修复：end_of_stack/control_stack 线程安全

这两个指针原来是全局变量，多个线程调用 `reset_machine()` 时互相覆盖导致 "stack overflow"。
改为 `thread_local`，每线程独立持有栈指针。`reset_machine()` 在每线程的 `init_vm_state()` 中调用。

## 三、问题解决记录

### 问题 1（已修复 ✅）：心跳对象未进入线程队列

**根因:** 三个并发问题叠加：
1. 线程启动与 `post()` 存在竞态——`event_base_loop` 尚未开始监听 wakeup fd
2. `fprintf` 无 `fflush` 导致日志丢失，无法定位问题
3. `thread_for_object()` 哈希错误——原始 `ptr * K % N` 因 8 字节对齐始终返回 bucket 0

**修复:**
- 添加 `std::atomic<bool> ready_` 标志 + spin-wait 同步启动/投递
- `post()` 中 EAGAIN/EWOULDBLOCK 正确处理
- 所有调试日志加 `fflush(stderr)`
- `thread_for_object()` 改为 `ptr / sizeof(object_t) % N` 按对象索引哈希

### 问题 2（已定位）：内存 swap 过大

**现象:** 30000 对象 × ~1.5KB/对象 = ~47MB 内存实际占用。GB 级 swap 来自 jemalloc arena 预分配。

**策略:** 已有 `max memory` 配置项。后续可配置 jemalloc dirty_decay 参数。

### 问题 3（已修复 ✅）：create_batch 中 0 延迟 call_out 阻塞游戏 tick

改为正延迟 call_out (`call_out("create_batch", 1)`)

### 问题 4（已修复 ✅）：T_FREED svalue double-free

**现象:** 30000 对象心跳执行时 `FATAL ERROR: T_FREED svalue freed. Previously freed by call_function_pointer`

**根因:** `apply_ret_value` 是全局 `svalue_t`，`call_function_pointer` 将结果写入它后返回 `&apply_ret_value`。8 线程同时调用时互相覆盖，`free_svalue` 释放了其他线程的结果。

**修复:** `apply_ret_value` → `thread_local`（apply.h/apply.cc/file.cc/checkmemory.cc）

### 问题 5（已修复 ✅）：debugmalloc 多线程竞争

**现象:** `debugmalloc: attempted to free non-malloc'd pointer` + `std::map` 红黑树损坏

**根因:** `MDmalloc`/`MDfree` 全局 hash table 和 `md_refjournal` map 无锁

**修复:** 加全局 mutex 保护 alloc/free 和 journal；临时关闭 `DEBUGMALLOC_EXTENSIONS`（非核心功能）

### 问题 6（已修复 ✅）：funptr_hdr_t::ref / program_t::func_ref 非原子

**现象:** 多线程创建 closure (`(: $1 > 1000 :)`) 时 ref count 竞争

**修复:** `funptr_hdr_t::ref` → `std::atomic<uint32_t>`，`program_t::func_ref` → `std::atomic<unsigned short>`，相关 varargs 调试打印加 `.load()`

### 问题 7（当前 🔴）：30000 对象心跳执行静默 segfault

**现象:** `process_heartbeats #1` 在 8 线程上成功触发，无 FATAL 错误，但执行期间进程静默死亡（libunwind backtrace 为空）

**已排除:** debugmalloc 竞争、apply_ret_value 竞争、funptr ref count 竞争、func_ref 竞争

**下一步:** ASAN 构建定位 use-after-free/buffer-overflow；或用 LLDB attach 获取有效 backtrace

## 四、下一步行动建议

### 优先级 1（当前）：定位最后一个 crash

**已确认：**
- `push_shared_string` → `make_shared_string` 修复了 "ref_string on non-shared string"
- macOS 无 CFRunLoop 线程上 GCD timer 死锁 → 改为 no-op timer（eval cost 计数器足够）
- `init_strings()` 在每个心跳线程调用（初始化 thread_local 字符串表）
- `svalue_to_int` + `sval_hash` 纯内容 hash（`whashstr`）

**当前 crash：** 非 ASAN 构建显示 `extend_string → debugrealloc → MDfree → abort`
- `MDfree` 无法在 debugmalloc hash table 中找到被释放的指针
- 可能: `extend_string` 被调用在 CONSTANT 字符串上，`MSTR_BLOCK` 给无效指针
- 可能: `debugrealloc` 在 `MDfree`/`MDmalloc` 之间 hash table 状态不一致

**行动：**
1. 在 `extend_string` 入口加 subtype 检查，拒绝非 MALLOC 字符串
2. 或在 `debugrealloc` 中将 `MDfree`+`realloc`+`MDmalloc` 放在同一个 mutex 锁内
3. 用 `lldb` 确认具体哪个字符串操作触发

### 优先级 2：提交核心修复

35 项测试通过后可先提交核心修复（不依赖 stress test 完成）。
