## 二、当前架构设计（最终版）

### 线程分布

```
主线程                         心跳线程池 (8 threads)
┌────────────────────┐         ┌──────────────────────────┐
│ 事件循环            │         │ Thread 0: heartbeat 队列  │
│ call_heart_beat()  │  post   │ Thread 1: heartbeat 队列  │
│  - 仅发送信号       │ ──────→ │ ...                      │
│ set_heart_beat()   │         │ Thread 7: heartbeat 队列  │
│  - post 到对应线程  │         │                          │
│ 玩家命令处理        │         │ process_heartbeats():     │
└────────────────────┘         │  合并队列 → 执行 LPC      │
                               └──────────────────────────┘
```

### 心跳执行流

1. `set_heart_beat(ob, to)` → `pool->thread_for_object(ob)->post(modify_heartbeat)` → 对象加入该线程的私有队列
2. `call_heart_beat()` → 每个线程 `post(process_heartbeats)` → 线程合并 next→main 队列 → 遍历 → 执行 LPC
3. 心跳门控 `g_heartbeat_gate`：创建阶段关闭（跳过 LPC），pool 启动后打开

### 关键修复：end_of_stack/control_stack 线程安全

这两个指针原来是全局变量，多个线程调用 `reset_machine()` 时互相覆盖导致 "stack overflow"。
改为 `thread_local`，每线程独立持有栈指针。`reset_machine()` 在每线程的 `init_vm_state()` 中调用。

## 三、当前遗留问题

### 问题 1（核心）：心跳对象未进入线程队列

**现象:** 30000 个对象创建完毕，`process_heartbeats()` 被调用，但 `queue=0`（队列为空）。
`hb avg=0 min=0 max=0` — 没有对象执行过一次 `heart_beat()`。

**已排查:**
- 门控 `g_heartbeat_gate` 已打开（gate=1 确认）
- `post()` 没有丢弃任务（无 "DROPPED" 日志）
- `process_heartbeats()` 确实被调用了（有日志）
- 但 `modify_heartbeat()` 从未被调用（无日志）
- `process_pending_tasks()` 也未被调用（无日志）

**推测根因:**
HeartbeatThread 的 `process_pending_tasks()` 是通过 `wakeup_cb` 触发的。
但 `fprintf(stderr)` 输出被缓冲——实际日志可能丢失了。
需要加 `fflush(stderr)` 或改用 `write(2, ...)` 确认。

**另一种可能:**
`event_base_loop(EVLOOP_ONCE)` 在 kqueue 上等待时，wakeup socket 的写入已经发生在线程进入 event_loop 之前。
此时 socket 缓冲区已有数据，但 kqueue 可能没有正确触发 wakeup_cb。
需要在线程进入 event_loop 后**再次写入**wakeup socket 来确保唤醒。

**解决思路:**
1. 在所有 fprintf 后加 `fflush(stderr)` 获取准确日志
2. 检查线程启动时序：`pool->start()` 返回时线程是否确实已进入 event_loop
3. 在线程 event_loop 入口处加一个 "ready" 标志（atomic），并让 `post()` 等待该标志
4. 如果 wakeup socket 有竞态，改用 `event_active()` 直接激活 wakeup event

### 问题 2：内存 swap 过大

**现象:** 30000 对象 × ~1.5KB/对象 = ~47MB 内存，但用户观察到数 GB swap。

**分析:**
- 对象内存线性增长正常（每 1000 对象 +~1.5MB）
- 配置已设 `clean_up : 0` 和 `swap : 0`
- GB 级的 swap 可能来自 jemalloc arena 预分配
- 或来自 clone_object 时的临时编译/字符串分配

**解决思路:**
- 添加 jemalloc `mallctl` 查询，区分"已分配"vs"常驻"内存
- 设置 jemalloc 的 `dirty_decay_ms` 和 `muzzy_decay_ms` 限制 arena 保留
- 使用 `max memory` 配置，在接近上限时主动清理

### 问题 3：create_batch 中 0 延迟 call_out 阻塞游戏 tick

**现象:** 大对象创建期间，0 延迟 `call_out` 循环阻塞游戏 tick，心跳无法分发。

**已修复:** 改为正延迟 call_out (`call_out("create_batch", 1)`)
